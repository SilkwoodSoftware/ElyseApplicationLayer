# Elyse Application - Developer Build Guide

## Purpose
This guide is for **developers** who need to build the Elyse application and prepare deployment packages. If you are a system administrator or installer looking to deploy the application to a production server, see the `README.txt` file included in each deployment package.

---

## Overview

The Elyse application consists of two independent components:
1. **Angular Frontend** - Static web application
2. **ASP.NET Core Backend** - REST API with SQL Server access

These components are built separately and can be deployed to different servers.

## Part 2: Building the Frontend

### 2.1 Frontend Versioning

The frontend uses **semantic versioning** with automatic build ID generation:

**Version Number (Manual)**:
- Stored in `elyse_app-frontend/package.json` as `"version": "1.0.0"`
- Format: `MAJOR.MINOR.PATCH` (Semantic Versioning)
- Updated manually by developers when releasing new versions
- Example: `"version": "1.2.3"`

**Build ID (Automatic)**:
- Automatically generated during build process
- Format: `YYYYMMDDHHmmss` (timestamp)
- Stored in `elyse_app-frontend/package.json` as `"buildId"`
- Example: `"buildId": "20260217143045"`
- Generated by `Build-Frontend.ps1` script before compilation

**Updating the Version Number**:
```powershell
# Edit elyse_app-frontend/package.json
# Change the "version" field manually:
{
  "name": "Elyse® Application Frontend",
  "version": "1.2.3",  # <-- Update this manually
  "buildId": "PLACEHOLDER_BUILD_ID",  # <-- Auto-generated during build
  ...
}
```

**Viewing Version Information**:
- After deployment, users can view version info via: **Information Menu → Frontend Nameplate**
- Displays: Application Name, Version, and Build ID

### 2.2 Frontend Configuration

The frontend loads the backend API URL from `assets/config.json` at application startup. This configuration file can be edited after deployment without rebuilding the application.

**Development configuration** (`elyse_app-frontend/src/assets/config.json`):
```json
{
  "apiUrl": "http://localhost:5021/api"
}
```

**Production configuration** (automatically set during package creation):
```json
{
  "apiUrl": "http://localhost:5000/api"
}
```

The backend URL can be changed by editing `assets/config.json` in the deployed application.

### 2.3 Build Using Build Script

```powershell
# Build and create deployment package
.\Build-Frontend.ps1 -CreatePackage
```

The build script performs the following operations:
1. Verifies Node.js and npm installation
2. Installs npm dependencies
3. **Generates and injects Build ID** into package.json (timestamp-based)
4. Builds production bundle
5. Updates config.json to production defaults
6. Creates deployment package in `.\build-output\`

**Note**: The Build ID is automatically generated as `YYYYMMDDHHmmss` and written to `package.json` before the build, ensuring each build has a unique identifier.

## Part 3: Security Architecture and Platform Constraints

### 3.1 ORIGINAL_LOGIN() and Windows Authentication Requirement

**CRITICAL:** The Elyse application's security architecture is fundamentally based on SQL Server's `ORIGINAL_LOGIN()` function, which has important implications for platform support and deployment options.

#### How the Security Architecture Works

1. **User Authentication**: Users authenticate to the backend using Windows credentials
2. **Database Connection**: Backend connects to SQL Server using Windows Authentication (Integrated Security)
3. **Application Roles**: Backend activates an application role (CONFIGURATOR, READER, REVIEWER, CONTROLLER, EDITOR, AUTHORISER) using `sp_setapprole`
4. **User Identification**: Database stored procedures use `ORIGINAL_LOGIN()` to identify the actual user
5. **SID Resolution**: The database resolves the Windows login to a Security Identifier (SID)
6. **Access Control**: Database enforces user-level ACLs based on the SID associated with `ORIGINAL_LOGIN()`

#### Why ORIGINAL_LOGIN() and SID-Based ACLs are Critical

`ORIGINAL_LOGIN()` is a SQL Server function that returns the original login name that connected to the database, even after switching to an application role via `sp_setapprole`.

**The database uses SID-based ACLs**, not username-based ACLs. This means:

1. **ORIGINAL_LOGIN() returns the Windows login** (e.g., `DOMAIN\JohnDoe`)
2. **SQL Server resolves this to a Windows SID** (Security Identifier, e.g., `S-1-5-21-...`)
3. **Database ACL tables store SIDs**, not usernames
4. **Stored procedures compare the user's SID against ACL SIDs** to determine access

This architecture allows the application to:

- **Enforce user-level permissions**: Each user sees only the data they have access to based on their SID
- **Maintain audit trails**: All database operations are attributed to the actual user's SID
- **Support multi-user SERVER environments**: With KCD, multiple users can share the same application role but have different data access based on their individual SIDs
- **Handle username changes**: If a user's name changes, their SID remains the same, preserving access rights

#### Platform Constraints Imposed by ORIGINAL_LOGIN()

**ORIGINAL_LOGIN() requires Windows Authentication**, which imposes the following constraints:

**For SERVER Deployments (USE_KCD=true):**
- ✅ **Windows Server required**: Backend must run on Windows Server with IIS
- ✅ **Active Directory required**: Users must authenticate via Windows domain
- ✅ **SQL Server on Windows required**: Windows Authentication for KCD
- ✅ **Windows workstations required**: Users must connect from domain-joined machines
- ❌ **Cannot use SQL Authentication**: Would break `ORIGINAL_LOGIN()` user identification and SID-based ACLs
- ❌ **Cannot run on Linux/macOS**: KCD is Windows-only

**For WORKGROUP Deployments (USE_KCD=false):**
- ✅ **Windows 10/11 required**: Backend runs as Windows Service
- ✅ **SQL Server with Windows Authentication**: Backend connects as user account
- ✅ **Single user only**: `ORIGINAL_LOGIN()` returns the service account (one user per machine)
- ❌ **Cannot use SQL Authentication**: Would break `ORIGINAL_LOGIN()` user identification
- ❌ **Cannot run on Linux/macOS**: Windows Authentication not available
- ❌ **Cannot provide user isolation**: Multiple users on same machine can access each other's backends

#### Why Cross-Platform Deployment is Not Possible (for Most Endpoints)

While .NET Core is cross-platform, the Elyse application cannot be fully deployed on Linux or macOS because:

1. **Windows Authentication dependency**: `ORIGINAL_LOGIN()` only works with Windows Authentication (Integrated Security)
2. **SID resolution requires Windows**: SQL Server can only resolve Windows logins to SIDs; non-Windows platforms cannot provide Windows SIDs
3. **SQL Authentication breaks security**: Using SQL Authentication would cause `ORIGINAL_LOGIN()` to return the SQL login name (e.g., 'sa'), not the actual user, and SQL logins do not have Windows SIDs
4. **Application roles require ORIGINAL_LOGIN()**: The entire security model depends on identifying the real user after switching to an application role
5. **Database ACLs are SID-based**: All ACL tables store Windows SIDs; without Windows Authentication, there is no SID to compare against
6. **Most stored procedures use ORIGINAL_LOGIN()**: The majority of stored procedures use `ORIGINAL_LOGIN()` to get the Windows login, resolve it to a SID, and check ACLs

**Attempting to use SQL Authentication or non-Windows platforms would result in:**
- All users appearing as the same SQL login to the database
- No Windows SID available for ACL checks
- Complete breakdown of user-level access control
- Loss of audit trail (all actions attributed to SQL login)
- Security vulnerabilities (users could access data they shouldn't see)
- Database queries failing when trying to resolve SIDs

#### Exception: Public Read-Only Endpoints

**Some READER role stored procedures do not use `ORIGINAL_LOGIN()` or perform ACL checks** because they are deemed to be public information. For these specific endpoints:

- ✅ **Cross-platform application layer is theoretically possible**: A Linux/macOS backend could connect using SQL Authentication
- ✅ **No Windows Authentication required**: These stored procedures don't call `ORIGINAL_LOGIN()`
- ✅ **No SID resolution needed**: No ACL checks means no Windows SID lookups

**However, this approach has significant limitations:**

- ⚠️ **Backend modification required**: The current backend uses Windows Authentication for all connections; it would need to be modified to support SQL Authentication for specific endpoints
- ⚠️ **Partial functionality only**: Only the subset of READER endpoints without ACL checks would work
- ⚠️ **Security boundary unclear**: Each stored procedure would need to be examined to establish which ones do not apply ACL checks.
- ⚠️ **Mixed authentication not supported**: The current architecture does not support mixing Windows Authentication and SQL Authentication
- ⚠️ **Not officially supported**: This configuration has not been tested or documented

**Recommendation**: For production deployments, use Windows-based infrastructure for the full application. Cross-platform deployment of public read-only endpoints is a theoretical possibility but would require development and testing effort.

#### Summary

The Elyse application is **Windows-only by design** due to its reliance on `ORIGINAL_LOGIN()` and Windows Authentication. This is not a limitation of the build scripts or .NET Core, but a fundamental architectural requirement for the security model to function correctly.

## Part 4: Building the Backend

### 4.1 Understanding Build Types

The Elyse backend supports **two different build types** for different deployment scenarios:

#### Framework-Dependent Build (for SERVER Deployment with Multi-User Support)

**Use for:** Windows Server with IIS, Active Directory, and Kerberos Constrained Delegation (KCD)

**Why framework-dependent is REQUIRED for multi-user SERVER environments:**
- **Windows Authentication Integration:** Framework-dependent builds integrate seamlessly with IIS's Windows Authentication module, allowing the backend to access the authenticated user's `WindowsIdentity` from the HTTP context
- **Kerberos Constrained Delegation (KCD):** The backend uses `WindowsIdentity.RunImpersonated()` to impersonate authenticated users when connecting to SQL Server, enabling the "double-hop" authentication required for user-level ACLs
- **ORIGINAL_LOGIN() Requirement:** The database uses `ORIGINAL_LOGIN()` to identify individual users for access control and auditing. KCD ensures SQL Server sees each connection as the actual user (e.g., `DOMAIN\JohnDoe`), not the service account
- **IIS Integration:** Framework-dependent builds work correctly with IIS's ASP.NET Core Module for process management and Windows Authentication

**Technical Details:**
- Produces a DLL (not standalone EXE)
- Requires .NET 8.0 Runtime on target server
- Uses web.config for IIS configuration
- Smaller package size (~28 MB)

#### Single-File Self-Contained Build (for WORKGROUP/Single-User Deployment)

**Use for:** Single user on Windows 10/11 workstation in workgroup (non-domain) environment

**Why single-file self-contained works for single-user deployment:**
- **No KCD Required:** Workgroup deployments use `USE_KCD=false`, so the backend connects to SQL Server as the user's account (no impersonation needed)
- **No .NET Runtime Dependency:** All .NET runtime files are embedded in a single executable, simplifying deployment
- **Standalone Executable:** Runs as a Windows Service without requiring .NET installation
- **Simple Deployment:** One .exe file to deploy - no need to install or maintain .NET Runtime on the target machine

**Technical Details:**
- Produces a single standalone .exe file with all dependencies embedded
- No .NET Runtime required on target server
- File size: ~100+ MB (single file)
- Ideal for workgroup/single-user scenarios

**Why self-contained CANNOT be used for SERVER deployments with multiple users:**

Self-contained builds bundle their own copy of the .NET runtime, which creates compatibility issues with Windows Authentication and KCD:

1. **IIS Windows Authentication Module Incompatibility:** IIS's Windows Authentication module is designed to work with the system-installed .NET runtime. Self-contained applications use their own bundled runtime, which may not integrate correctly with IIS's authentication pipeline, preventing the backend from accessing the authenticated user's `WindowsIdentity`

2. **KCD Token Delegation Issues:** Kerberos Constrained Delegation relies on Windows security infrastructure that expects applications to use the system's .NET runtime. The bundled runtime in self-contained builds may not properly handle Kerberos ticket delegation, breaking the double-hop authentication flow

3. **WindowsIdentity.RunImpersonated() Behavior:** This API requires deep integration with Windows security subsystems. Self-contained runtimes may have different interop behavior that prevents proper user impersonation when opening SQL connections

4. **ASP.NET Core Module Integration:** IIS's ASP.NET Core Module (ANCM) is optimized for framework-dependent applications. Self-contained applications may not receive the authenticated user context correctly through the ANCM pipeline

**Result:** In SERVER environments with multiple users, self-contained builds would cause SQL Server to see all connections as the service account instead of individual users, breaking user-level ACLs and audit trails that depend on `ORIGINAL_LOGIN()`.

### 3.3 Manual Build for Self-Contained Single-File (WORKGROUP Deployment)

For WORKGROUP deployments, create a single-file self-contained executable:

```powershell
# Navigate to backend directory
cd elyse_asp-backend

# Restore packages
dotnet restore

# Build and publish (single-file self-contained for workgroup)
dotnet publish -c Release -r win-x64 --self-contained true `
    -p:PublishSingleFile=true `
    -p:IncludeNativeLibrariesForSelfExtract=true `
    -o ..\build-output\elyse-backend-workgroup

# Return to root
cd ..
```

**Build Options Explained:**
- `--self-contained true` - Includes .NET runtime in the output
- `-p:PublishSingleFile=true` - Bundles everything into a single .exe file
- `-p:IncludeNativeLibrariesForSelfExtract=true` - Includes native libraries in the single file

**Note:** A helper script [`Build-Backend-Workgroup.ps1`](Build-Backend-Workgroup.ps1) is provided in the project root that automates the build cleanup and packaging process. You can use it instead of the manual steps below.

#### Create WORKGROUP Deployment Package

```powershell
# Create deployment package
$WorkgroupPackage = "Elyse-Backend-Deploy-WORKGROUP-$(Get-Date -Format 'yyyyMMdd-HHmmss')"
$buildOutputDir = ".\build-output"

# Ensure build output directory exists
if (-not (Test-Path $buildOutputDir)) {
    New-Item -ItemType Directory -Path $buildOutputDir -Force | Out-Null
}

# Clean up unnecessary files from build output
# Single-file publish creates extra files that aren't needed for deployment
Remove-Item "$buildOutputDir\elyse-backend-workgroup\*.pdb" -ErrorAction SilentlyContinue
Remove-Item "$buildOutputDir\elyse-backend-workgroup\appsettings.json" -ErrorAction SilentlyContinue
Remove-Item "$buildOutputDir\elyse-backend-workgroup\web.config" -ErrorAction SilentlyContinue
Remove-Item "$buildOutputDir\elyse-backend-workgroup\LICENSE" -ErrorAction SilentlyContinue

# Create temporary package directory
New-Item -ItemType Directory -Path "$buildOutputDir\$WorkgroupPackage" -Force | Out-Null

# Copy single-file executable
Copy-Item -Path "$buildOutputDir\elyse-backend-workgroup\elyse_asp-backend.exe" `
          -Destination "$buildOutputDir\$WorkgroupPackage\"

# Create .env template for workgroup
@"
# .env Configuration for Workgroup Environment
# IMPORTANT: Set USE_KCD=false for workgroup deployments

# Database Configuration
DB_TYPE=mssql
DB_HOST=YOUR_SQL_SERVER_NAME_OR_IP
DB_PORT=1433
DB_NAME=Elyse_DB
DB_SYNCHRONIZE=false
DB_ENCRYPT=true
DB_TRUST_SERVER_CERTIFICATE=true

# Authentication Mode
# CRITICAL: Must be false for workgroup (no Windows Authentication/KCD)
USE_KCD=false

# Server Configuration
# Kestrel will listen on this URL
SERVER_URLS=http://0.0.0.0:5000

# Application Role Passwords (from SQL Server)
CONFIGURATOR_PASSWORD=
READER_PASSWORD=
REVIEWER_PASSWORD=
CONTROLLER_PASSWORD=
EDITOR_PASSWORD=
AUTHORISER_PASSWORD=
"@ | Out-File "$buildOutputDir\$WorkgroupPackage\.env.template" -Encoding UTF8

# Create README from WORKGROUP_README.txt template
$readmeTemplatePath = ".\WORKGROUP_README.txt"
$buildDate = Get-Date -Format 'yyyy-MM-dd HH:mm:ss'
$readmeContent = Get-Content $readmeTemplatePath -Raw
$readmeContent = $readmeContent -replace '\{\{VERSION\}\}', $version
$readmeContent = $readmeContent -replace '\{\{BUILD_DATE\}\}', $buildDate
$readmeContent | Out-File "$buildOutputDir\$WorkgroupPackage\README.txt" -Encoding UTF8 -Force

# Create ZIP file
$zipPath = Join-Path $buildOutputDir "$WorkgroupPackage.zip"
Compress-Archive -Path "$buildOutputDir\$WorkgroupPackage\*" `
                 -DestinationPath $zipPath -Force

# Clean up temporary directory
Remove-Item "$buildOutputDir\$WorkgroupPackage" -Recurse -Force

Write-Host "`nWORKGROUP deployment package created: $zipPath" -ForegroundColor Green
Write-Host "Package type: Single-file self-contained (~100+ MB)" -ForegroundColor Yellow
Write-Host "Contents: elyse_asp-backend.exe + .env.template + 01_DOMAIN_SETUP.txt + 03_KCD_CONFIGURATION.txt" -ForegroundColor Yellow
Write-Host "Target: Windows 10/11 Workgroup environments" -ForegroundColor Yellow
Write-Host "NO .NET Runtime required on target machine" -ForegroundColor Green
```

## Part 8: Common Build Issues

### Frontend Build Fails

**Error:** `npm: command not found`
- **Solution:** Install Node.js and npm

**Error:** `Cannot find module`
- **Solution:** Run `npm install` first

**Error:** Frontend cannot connect to backend
- **Solution:** Edit `assets/config.json` in deployed application (no rebuild required)

### Backend Build Fails

**Error:** `.NET SDK not found`
